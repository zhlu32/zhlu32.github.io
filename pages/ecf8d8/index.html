HTTP参数污染-HPP

# 漏洞介绍

HTTP参数污染（HTTP Parameter Pollution, or HPP）特指在HTTP请求中，站点是如何处理它收到的请求参数的。当请求参数被篡改，但站点却信任它， 导致站点的非预期行为，此时HTTP参数污染漏洞就发生了。

该漏洞分为，对你不可见发生在服务端的Server-Side HPP，和对你可见发生在客户端的Client-Side HPP。

# Server-Side HPP

假设银行处理转账的业务URL，https://www.bank.com/transfer?from=12345&to=67890&amount=5000，它通过三个参数from, to, and amount， 控制从12345账号(from)转出5000元(amount)到67890账号(to)。

在URL的尾部再拼接一个重复的from属性值，https://www.bank.com/transfer?from=12345&to=67890&amount=5000&from=ABCDEF，如果Server存在 HPP漏洞，取from的值为ABCDEF，会导致从非自己的ABCDEF账号转出5000元到67890账号。

为了修复上面的漏洞，银行开发人员在URL中去除了from字段，https://www.bank.com/transfer?to=67890&amount=5000，通常Server端的代码我们是看 不到的，假如我偷到了银行的Server代码，实现如下：

user.account = 12345

def prepare_transfer(params) 
  params << user.account 
  transfer_money(params) #user.account (12345) becomes params[2] 
end

def transfer_money(params)
  to = params[0]
  amount = params[1] 
  from = params[2] 
  transfer(to,amount,from) 
end

查看代码逻辑后，可以通过直接在URL后添加&from=ABCDEF，请求走到trabsfer_money()函数后，params数组值为[67890,5000,ABCDEF,12345]， 同样的产生了修复前的漏洞。

# Client-Side HPP

Client-Side HPP漏洞，首先注入参数到URL，紧跟着影响到返回给用户的页面内容。

URL http://host/page.php?par=123 对应的后端实现逻辑如下：

<? $val=htmlspecialchars($_GET['par'],ENT_QUOTES); ?>
<a href="/page.php?action=view&par='.<?=$val?>.'">View Me!</a>

上面的URL实现了查看编号为123的文章内容。

假如par参数值传入为123%26action=edit，跳转的href值被篡改为/page.php?action=view&par=123&aciton=edit，此时如果存在HPP漏洞， 用户看到的页面将发生改变。

# HackerOne Social Sharing Buttons (opens new window)

HackerOne的Blog文章支持分享到Facebook、Twitter等社交平台，分享到社交平台的链接，其它用户点击后，可以跳转回HackOne的Blog页面。

假设HackerOne的某篇Bloghttps://hackerone.com/blog/introducing-signal，此时把该文章分享到facebook，在分享的url后添加?&u=https://vk.com/durov， 则分享到facebook的url为 https://www.facebook.com/sharer.php?u=https://hackerone.com/blog/introducing-signal?&u=https://vk.com/durov，当其它 用户点击该链接后，并不能跳回到原分享的Blog链接，而是跳转到https://vk.com/durov。

# 总结

HPP漏洞的危害取决于后端站点如何处理被污染的参数，这些逻辑对于hacker人员通常是黑盒，需要hacker根据经验查看后端传入被污染 参数后的返回结果的变化，猜测它的处理逻辑。
